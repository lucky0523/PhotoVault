# 回收站功能实施方案（含客户端状态同步）

## Context

当前删除是物理删除（DELETE + unlink），误删不可恢复。且客户端不记录"已备份"状态，FolderDetailScreen 的单张照片状态是伪造的（前 N 张标对勾）。

本方案实现完整的回收站功能：服务端软删除 + 彻底删除标记，状态同步给客户端，客户端维护四种照片状态（未备份/已备份/回收站中/已删除），自动备份跳过回收站和已删除项，手动可重传或从 Web 回收站还原。

## 四种状态定义

| 状态 | 服务端条件 | 客户端表现 |
|------|-----------|-----------|
| 已备份 (active) | `deleted_at IS NULL AND purged_at IS NULL` | 绿色对勾 |
| 回收站中 (trashed) | `deleted_at IS NOT NULL AND purged_at IS NULL` | 橙色标记 + 剩余时间 |
| 已删除 (purged) | `purged_at IS NOT NULL` | 红色标记 |
| 未备份 (not_backed_up) | 服务端无记录 | 无标记 |

## 物理移动方案（重要补充）

> **修订说明**：原方案中"删除仅标记 `deleted_at`，不移动物理文件"已变更为：删除时将物理文件移动到 `.trash` 目录，恢复时移回原位置，彻底删除时从 `.trash` 删除物理文件。此变更确保文件存储与回收站状态的视觉一致性。

### 存储结构设计

```
{storage_root}/{username}/
├── {device_name}/                    # 正常文件存储
│   └── Pictures/ithome/photo.png
├── .trash/                           # 回收站目录（按原路径结构）
│   └── {device_name}/
│       └── Pictures/ithome/photo.png
└── .locks/                           # 文件锁目录
    └── {file_hash}.lock
```

**设计原则**：
- `.trash` 目录保持与原存储相同的目录结构，便于追溯原位置
- 文件夹删除时，其下所有文件按相对路径结构移动到 `.trash/{device_name}/...`
- 恢复时按相同结构移回原位置

### 文件锁机制

**目的**：防止大量/大文件移动时，多处同时操作导致文件与数据库不一致。

**实现方式**：
- **锁类型**：基于 `fcntl.flock` 的文件级排他锁（`LOCK_EX | LOCK_NB`）
- **锁路径**：`{storage_root}/{username}/.locks/{file_hash}.lock`
- **超时策略**：非阻塞模式（`LOCK_NB`），获取失败立即返回 "File is currently being processed"
- **生命周期**：`try/finally` 保证锁释放

**核心方法**（`server/app/services/file_browse_service.py`）：

```python
def _get_lock_path(self, username: str, file_hash: str) -> str:
    """获取锁文件路径"""
    lock_dir = f"{self._storage_root}/{username}/.locks"
    os.makedirs(lock_dir, exist_ok=True)
    return f"{lock_dir}/{file_hash}.lock"

def _acquire_lock(self, username: str, file_hash: str) -> Optional[int]:
    """获取文件锁，非阻塞模式"""
    lock_path = self._get_lock_path(username, file_hash)
    fd = os.open(lock_path, os.O_RDWR | os.O_CREAT, 0o644)
    fcntl.flock(fd, fcntl.LOCK_EX | fcntl.LOCK_NB)
    self._lock_files[file_hash] = fd
    return fd

def _release_lock(self, file_hash: str) -> None:
    """释放文件锁"""
    fd = self._lock_files.pop(file_hash, None)
    if fd is not None:
        fcntl.flock(fd, fcntl.LOCK_UN)
        os.close(fd)
```

### 核心操作流程

#### 1. 删除文件（移动到.trash）

```
用户请求 DELETE /files/{file_id}
  → 验证权限
  → 获取文件锁 (file_hash)
  → 物理移动文件到 .trash/{device_name}/...
  → 更新数据库: file_path = 新路径, deleted_at, deleted_batch_id
  → 提交事务
  → 释放锁
```

**关键代码**：
```python
async def delete_file(self, user_id: int, file_id: int) -> tuple[bool, str]:
    file_info = await self.get_file_info(user_id, file_id)
    # ...
    lock_acquired = self._acquire_lock(username, file_info.file_hash)
    if not lock_acquired:
        return False, "File is currently being processed"
    try:
        new_path = self._move_to_trash(username, file_info.device_name, file_info.file_path)
        # 更新数据库
        await self._db.execute("""
            UPDATE file_records
            SET deleted_at = ?, deleted_batch_id = ?, file_path = ?
            WHERE id = ? AND user_id = ?
        """, (now, batch_id, new_path, file_id, user_id))
        await self._db.commit()
        return True, "File moved to trash"
    finally:
        self._release_lock(file_info.file_hash)
```

#### 2. 恢复文件（从.trash移回）

```
用户请求 POST /files/trash/{file_id}/restore
  → 验证权限
  → 获取文件锁 (file_hash)
  → 物理移动文件回原位置
  → 更新数据库: file_path = 原路径, deleted_at = NULL, deleted_batch_id = NULL
  → 提交事务
  → 释放锁
```

#### 3. 彻底删除（从.trash删除）

```
用户请求 DELETE /files/trash/{file_id}
  → 验证权限
  → 获取文件锁 (file_hash)
  → 标记 purged_at
  → 物理删除 .trash 中的文件
  → 清理空目录
  → 提交事务
  → 释放锁
```

### 并发安全保障

1. **文件锁**：同一文件的多个操作串行化，防止并发冲突
2. **引用处理**：引用文件（`is_reference=true`）仅修改数据库标记，不移动物理文件
3. **事务提交**：文件移动和数据库更新在同一事务中，失败时回滚
4. **空目录清理**：删除后自动清理空目录，保持存储整洁

## 关键设计决策

1. **彻底删除不 DELETE 记录，而是标记 `purged_at`**：保留记录让客户端能同步到"已删除"状态。purged 记录永久保留在数据库中，不清理，确保客户端任何时候连接都能同步到已删除状态。
2. **purge 时删物理文件但保留记录**：purged 记录的 file_path 等字段保留，供客户端按 file_hash 匹配。
3. **/backup/check 扩展返回 status**：客户端备份时即时获知文件状态。trashed/purged 返回 is_duplicate=false，允许手动重传。
4. **新增 /files/status-sync 接口**：客户端启动/备份前拉取所有非 active 状态文件的 hash+状态，批量同步本地状态表。
5. **手动重传单张**：trashed/purged 状态的照片在客户端长按菜单提供"重新备份"，重传成功后服务端清除删除标记。自动备份和"立即备份"均跳过这些照片。
6. **回收站还原在 Web 端**：Web 回收站页面提供恢复按钮，恢复后状态变 active，客户端下次同步感知到。

- `_get_trash_path(user_id, device_name)` — 获取 `.trash/{device_name}` 目录路径
- `_move_to_trash(username, device_name, file_path)` — 物理移动文件到 `.trash`
- `_restore_from_trash(username, device_name, file_path)` — 从 `.trash` 移回原位置
- `_delete_from_trash(file_path)` — 从 `.trash` 删除物理文件
- `_cleanup_empty_directories(file_path)` — 清理空目录
- `_acquire_lock(username, file_hash)` — 获取文件锁（非阻塞）
- `_release_lock(file_hash)` — 释放文件锁

- `purge_all(user_id)` — 批量 purge
- `auto_purge_expired(expiry_days)` — 回收站超 `trash_retention_days` 配置天数的设 purged_at（记录保留，仅删物理文件）
- `get_status_changes(user_id)` — 返回所有 trashed/purged 文件的 {file_hash, status, deleted_at, purged_at}，供客户端同步
